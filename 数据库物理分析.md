# 实验室管理系统数据库物理分析

基于提供的用户实体、实验室实体、预约记录实体以及检修记录实体的描述，我们可以进行数据库物理分析。这里，我将为每个实体提出可能的物理数据模型设计，包括表结构、字段类型、主键、外键以及可能的索引设计。

## （1）User 表

该表主要根据实验室管理系统用户的实体进行分析、设计、并创建，其中除了包含用户注册输入的基本信息外，还包含用户的角色，该字段主要用于区分系统管理员、实验室管理员、教师和学生等不同角色，系统会根据不同用户跳转至对应页面。本表将主键索引给到唯一性较强的用户 ID 列，且该列运用 AUTO_INCREMENT 定义自增序列保障每个新用户都有自己的专属 ID。另外由于用户需要用该列作为账号登录本系统，故对此列加入了唯一性索引约束。

### 表 4-1 tb_user 表

| 字段名 | 字段类型 | 描述 |
|--------|----------|------|
| user_id | INT PRIMARY KEY AUTO_INCREMENT | 用户 ID（主键） |
| user_account | VARCHAR(50) NOT NULL UNIQUE | 用户账号 |
| password | VARCHAR(255) NOT NULL | 密码（当前项目为明文存储；生产环境应加密） |
| user_name | VARCHAR(50) | 用户姓名 |
| role | VARCHAR(20) NOT NULL | 用户角色（学生/教师/实验室管理员/系统管理员） |
| avatar | LONGTEXT | 头像（Base64或长 URL；使用 LONGTEXT 避免截断） |

## （2）Lab 表

该表有实验室管理系统实验室表实体进行分析、设计并创建，其中由于实验室编号有较强的唯一性的特点，故主键索引设置在 ID 列，除 ID 外本表还包含实验室编号、实验室名称、所属分类、管理员、位置、开放时间、关闭时间、设备清单、状态、描述以及图片URL等基本信息，该表主要用于存储实验室的基本信息。

### 表 4-2 lab 表

| 字段名 | 字段类型 | 描述 |
|--------|----------|------|
| id | BIGINT PRIMARY KEY AUTO_INCREMENT | 实验室 ID（主键） |
| lab_code | VARCHAR(50) NOT NULL UNIQUE | 实验室编号 |
| name | VARCHAR(100) | 实验室名称 |
| category_id | BIGINT NOT NULL | 分类 ID（外键） |
| manager_id | INT NOT NULL | 实验室管理员用户 ID（外键，关联 tb_user.user_id） |
| manage_id | INT NOT NULL | **历史兼容列**：误拼字段，与 manager_id 同值（建议数据库侧删除，仅保留 manager_id） |
| location | VARCHAR(100) | 位置 |
| open_time | TIME NOT NULL | 开放时间 |
| close_time | TIME NOT NULL | 关闭时间 |
| equipment | VARCHAR(500) | 设备清单 |
| status | INT NOT NULL | 状态（0-停用，1-正常，2-维护中） |
| description | VARCHAR(255) | 描述 |
| image_url | VARCHAR(255) | 图片URL |

## （3）Reservation 表

该表基于实验室管理系统的预约记录实体分析进行设计，其中 ID 列作为预约编号，设为主键且包含自增元素。user_id 则是预约用户的 ID，该列可便于追溯到用户的预约情况，有利于预约管理。其他信息主要为预约操作的基本信息如预约单号、实验室ID、开始时间、结束时间、预约用途、预约状态、使用状态、拒绝原因、审核人、审核时间等。

### 表 4-3 reservation 表

| 字段名 | 字段类型 | 描述 |
|--------|----------|------|
| id | BIGINT AUTO_INCREMENT PRIMARY KEY | 预约 ID（主键） |
| order_no | VARCHAR(32) NOT NULL UNIQUE | 预约单号 |
| user_id | INT | 用户 ID（外键，关联 tb_user.user_id） |
| lab_id | BIGINT | 实验室 ID（外键，关联 lab.id） |
| start_time | TIMESTAMP NOT NULL | 开始时间 |
| end_time | TIMESTAMP NOT NULL | 结束时间 |
| purpose | VARCHAR(255) | 预约用途 |
| status | INT NOT NULL | 预约状态（0-待审核，1-已通过，2-已拒绝，4-已取消） |
| use_status | INT NOT NULL | 使用状态（0-待使用，1-使用中，2-已结束，4-已取消） |
| reject_reason | VARCHAR(255) | 拒绝原因 |
| audit_user_id | INT | 审核人 ID（外键） |
| audit_time | TIMESTAMP | 审核时间 |
| created_time | TIMESTAMP | 创建时间 |
| updated_time | TIMESTAMP | 更新时间 |

## （4）Repair 表

该表主要基于实验室管理系统的报修记录实体分析进行设计，其中报修 ID 设置为主键索引并加以自增，保障每个报修记录的唯一性。其次设计了外键 user_id 列以关联 User 表中的 user_id，以及 lab_id 列关联 Lab 表中的 id，便于数据检索以及数据更新。其他列则主要为报修图书的报修信息以及申请信息构成。

### 表 4-4 repair 表

| 字段名 | 字段类型 | 描述 |
|--------|----------|------|
| id | BIGINT AUTO_INCREMENT PRIMARY KEY | 报修 ID（主键） |
| user_id | INT | 报修人 ID（外键，关联 tb_user.user_id） |
| lab_id | BIGINT | 实验室 ID（外键，关联 lab.id） |
| reservation_id | BIGINT | 关联预约 ID（外键） |
| title | VARCHAR(100) NOT NULL | 报修标题 |
| description | VARCHAR(500) | 报修描述 |
| status | INT NOT NULL | 报修状态（0-待处理，1-处理中，2-已完成，3-已关闭） |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |

## （5）Maintenance 表

该表基于实验室管理系统的检修记录实体分析进行设计，其中检修 ID 列为检修编号，用于记录检修信息以及每条检修记录的检修 ID，这样有利于对于每个检修的处理情况可以追溯到对应的检修情况，方便管理员管理检修记录，该列设为主键且自增。且表中 lab_id 设有外键可关联之 Lab 表下的 id 列，外键的加入极大的增加了数据检索的效率，也为数据的同步性做了根本的保障。其他信息主要为检修的基本信息。

### 表 4-5 maintenance 表

| 字段名 | 字段类型 | 描述 |
|--------|----------|------|
| id | BIGINT AUTO_INCREMENT PRIMARY KEY | 检修 ID（主键） |
| lab_id | BIGINT NOT NULL | 实验室 ID（外键） |
| repair_id | BIGINT | 关联的报修记录 ID（外键，关联 repair.id，可为空） |
| content | TEXT NOT NULL | 检修内容 |
| maintenance_unit | VARCHAR(100) | 检修单位 |
| maintenance_time | TIMESTAMP NOT NULL | 检修时间 |
| created_time | TIMESTAMP NOT NULL | 创建时间 |
| status | INT NOT NULL | 检修状态（0-处理中，1-已完成） |
| handler | VARCHAR(50) | 检修人 |
| handler_phone | VARCHAR(20) | 联系电话 |

## （6）LabCategory 表

该表基于实验室管理系统的实验室分类实体分析进行设计，其中分类 ID 设置为主键索引并加以自增，保障每个分类的唯一性。其次设计了外键 admin_id 列以关联 User 表中的 user_id，便于数据检索以及数据更新。其他列则主要为分类的基本信息构成。

### 表 4-6 lab_category 表

| 字段名 | 字段类型 | 描述 |

| name | VARCHAR(50) NOT NULL | 分类名称 |
| description | VARCHAR(255) | 分类描述 |
| manager_id | INT | 分类管理员用户 ID（外键，关联 tb_user.user_id） |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |

## （7）Announcement 表

该表基于实验室管理系统的公告实体分析进行设计，其中公告 ID 设置为主键索引并加以自增，保障每个公告的唯一性。其次设计了外键 publisher_id 列以关联 User 表中的 user_id，以及 lab_id 列关联 Lab 表中的 id（可为空，表示全局公告），便于数据检索以及数据更新。其他列则主要为公告的基本信息构成。

### 表 4-7 announcement 表

| 字段名 | 字段类型 | 描述 |
|--------|----------|------|
| id | BIGINT AUTO_INCREMENT PRIMARY KEY | 公告 ID（主键） |
| title | VARCHAR(100) NOT NULL | 公告标题 |
| content | TEXT NOT NULL | 公告内容 |
| publisher_id | INT | 发布人 ID（外键，关联 tb_user.user_id） |
| lab_id | BIGINT | 实验室 ID（外键） |
| status | INT NOT NULL | 公告状态（0-草稿，1-已发布，2-已撤回） |
| publish_time | TIMESTAMP | 发布时间 |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |

## 总结

实验室管理系统的数据库设计包含全部7个核心表：

1. **tb_user** - 用户表：存储系统用户基本信息和角色权限
2. **lab_category** - 实验室分类表：管理实验室分类信息
3. **lab** - 实验室表：存储实验室基本信息和配置
4. **reservation** - 预约记录表：管理用户实验室预约信息
5. **repair** - 报修记录表：记录设备故障报修信息
6. **maintenance** - 检修记录表：记录设备检修处理信息
7. **announcement** - 公告表：管理系统公告发布

每个表都设计了合理的主键、外键约束和索引，确保数据完整性和查询效率。通过外键关联实现了表间的数据一致性，为系统的稳定运行提供了数据层面的保障。

## 备注：关于 lab.manage_id（误拼字段）

当前项目的 `Lab` 实体同时存在 `manager_id`（标准字段）与 `manage_id`（历史误拼字段）两列兼容逻辑：

- **推荐数据库最终形态**：仅保留 `manager_id`，删除 `manage_id`。
- **当前代码兼容**：在保存/更新实验室管理员时，会将 `manage_id` 写成与 `manager_id` 同值，以避免旧库中两列同时 NOT NULL 时插入失败。
